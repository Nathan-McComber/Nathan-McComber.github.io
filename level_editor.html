Redesigning a Videogame Level Editor

As a freshman, I programmed a video game with 3 other students as a class project. One of my primary responsibilities was creating a level editor for the level designer of the game. As a personal project, I decided to apply what I have learned since freshman year to improve its design.

Goal: Iterate on the design of the level editor to make it simpler and easier to use.

Project Type:

Personal

Role:

Designer

Duration:

~ 8 hours

Original Editor Overview

A. Select the tile you want to place

B. Button that changes the tile's variant

C. Opens the “Edit Stats” window

D. Sets the currency based on the set numbers of items

Assessing the Original Editor

I performed a Heuristic Review by stepping through the workflow of making a level to assess the original editor.

Positives

A. Always display relevant information (Visibility of System Status)

B. Show all available tiles (Recognition Rather than Recall)

C. Show currently selected tile (Visibility of System Status)

D. Allow users to specify number of items rather than forcing them to calculate currency (Flexibility & Efficiency of Use)

General Positives:

  • The main focal point is the level itself (Aesthetic & Minimalist Design)
  • “Tiles” is a word from the industry that is reused (Match Between System & Real World)

Negatives

A. Poor text hierarchy - labels and values look the same (Aesthetic & Minimalist Design)

B. Outdated Graphic - it's a salt shaker in the final game (Match Between System & Real World)

C. 2 Slots for the same object - enabled/disabled (Aesthetic & Minimalist Design)

D. Forces a 2nd click (Flexibility & Efficiency of Use)

E. Disabled items can still be given values (Error Prevention)

F. No warning that changed this will affect the specified numbers of items (Error Prevention)

General Negatives:

  • Multiple entrances and exits can be placed - not possible in final game (Error Prevention)
  • No way to undo actions or reset values to defaults (Help Users Recognize, Diagnose, & Recover from Errors)
  • No indication that right click is used to remove tiles (Help & Documentation)

Early Concepts/Sketches

Before jumping into a digital space, I explored different ideas by making pencil and paper sketches.

Key Sketches

Put all of the level information on one screen. This gets rid of the “Edit Stats” screen and reduces clicks.

Have explicit dropdowns for tile properties/variants. This is much more clear than clicking on the tile's image in the original editor.

Pair item enabling with item count and prevent users from giving an item count if the item is disabled. This helps prevent errors.

Final Product

Major Changes

A. Give the ability to “Undo” and “Redo” (User Control & Freedom)

B. Highlight the currently selected tile (Visibility of System Status)

C. Provide tooltips to explain hotkeys (Help & Documentation)

D. Only allow enabled items to be given values (Error Prevention)

E. Give users the option to use item count OR choose a specific number for currency (User Control & Freedom)

F. Make it clear that tile properties can be edited AND show how the current tile looks (User Control and Freedom + Recognition Rather than Recall)

General Changes:

  • Added warning dialogs for unsaved changes and having multiple entrances/exits (Error Prevention)
  • All level information is on one screen - less clicks (Flexibility & Efficiency of Use)

Reflection

The new level editor fixes several issues that the original design had. It shows users all of the actions they can perform on one screen and makes the outcome of those actions obvious. It also provides hints with tooltips and prevents mistakes with warning dialogs.

Takeaways:

  • Inputs should be obvious and have a clear purpose.
  • Workflow steps should be kept in mind while designing.
